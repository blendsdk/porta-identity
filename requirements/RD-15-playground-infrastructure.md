## Feature Overview

This requirement covers the infrastructure needed to make the Playground Application
(RD-14) work out of the box. It includes an enhanced seed script that creates multiple
organizations with different 2FA policies, multiple test users with different 2FA
enrollment states, and pre-configured OIDC clients for each scenario.

It also provides a one-command startup (`yarn playground`) that orchestrates Docker
services, database migrations, seed data, the Porta server, and the playground app —
so anyone can go from `git clone` to a running, explorable Porta instance in under
5 minutes.

## Functional Requirements

- [ ] **Enhanced playground seed script** (`scripts/playground-seed.ts`) — Replaces the

  The script must be **idempotent** (safe to re-run).

  - Custom claim: `employee_id` (type: string)

- [ ] **One-command startup** — `yarn playground` from the project root that:
  1. Starts Docker services (`docker compose up -d`)
  2. Waits for Postgres and Redis to be healthy
  3. Runs database migrations
  4. Runs the enhanced seed script
  5. Starts the Porta server (background)
  6. Starts the playground static server (foreground)
  7. Prints a summary with URLs and test credentials

- [ ] **Teardown command** — `yarn playground:stop` that stops the Porta server and
  optionally stops Docker services.

### Should Have

- [ ] **Reset command** — `yarn playground:reset` that drops and recreates the database,
  re-runs migrations, and re-seeds. Useful when testing leaves the data in a bad state.

- [ ] **Seed summary output** — After seeding, print a formatted table of all created
  resources with IDs, emails, passwords, and client IDs.

- [ ] **Health check before seed** — Verify Docker services are running before
  attempting to seed data. Show helpful errors if Postgres/Redis are unreachable.

### Won't Have (Out of Scope)

- Production data seeding (this is dev-only playground data)
- Seed data for performance testing (many users/orgs)
- Seed data cleanup on Porta shutdown (data persists in Docker volumes)

## Technical Requirements

### Seed Script Architecture

The enhanced seed script (`scripts/playground-seed.ts`) follows the existing pattern:

1. Suppress logger noise (`LOG_LEVEL=fatal`)
2. Dynamic import of service modules
3. Connect to database and Redis
4. Run migrations
5. Create resources in dependency order: orgs → apps → clients → users → RBAC → 2FA enrollment
6. Write `playground/config.generated.js`
7. Print summary
8. Disconnect

### 2FA Enrollment During Seed

For users that need 2FA pre-enrolled:

- **Email OTP users**: Set `two_factor_enabled = true` and `two_factor_method = 'email'`
  in the user record. No additional setup needed — OTP codes are generated per-login.

- **TOTP users**: Call the TOTP setup service to generate a TOTP secret, encrypt it,
  and store it. The seed script should log the TOTP secret (base32) so the developer
  can add it to their authenticator app. The seed should also generate recovery codes
  and log them.

- **Fresh users (TOTP org)**: Leave `two_factor_enabled = false` so the org policy
  forces them through the setup flow on first login.

### Startup Script Architecture

The `yarn playground` command runs a shell script (`scripts/run-playground.sh`) that:

```bash
#!/bin/bash
set -e

# 1. Start Docker (if not already running)
docker compose -f docker/docker-compose.yml up -d

# 2. Wait for services
# ... health check loop for Postgres and Redis

# 3. Run seed (includes migrations)
yarn tsx scripts/playground-seed.ts

# 4. Start Porta in background
yarn dev &
PORTA_PID=$!

# 5. Wait for Porta to be healthy
# ... poll GET http://localhost:3000/health

# 6. Start playground
cd playground && yarn start

# 7. On exit, kill Porta
kill $PORTA_PID
```

### Config File Format

`playground/config.generated.js` uses ES module exports so the playground can import it:

```javascript
// @ts-nocheck — Auto-generated, do not edit
export const PLAYGROUND_CONFIG = { ... };
```

The file should be `.gitignore`d since it contains generated client IDs/secrets.

## Integration Points

### With RD-04 (Organizations)
- Creates 5 organizations with different settings and 2FA policies
- Uses `createOrganization()` service function

### With RD-05 (Clients)
- Creates public and confidential clients for each org
- Configures redirect URIs pointing to `http://localhost:4000/callback.html`

### With RD-06 (Users)
- Creates test users with various statuses (active, inactive, suspended)
- Sets passwords via the user service

### With RD-08 (RBAC & Custom Claims)
- Creates roles, permissions, and custom claims for the playground application
- Assigns roles to test users

### With RD-12 (2FA)
- Enrolls specific users in email OTP or TOTP 2FA
- Generates and logs TOTP secrets and recovery codes

### With RD-14 (Playground Application)
- Generates `playground/config.generated.js` consumed by the playground app
- Startup script orchestrates both Porta and playground together

## Scope Decisions

| Decision | Options Considered | Chosen | Rationale |
|----------|-------------------|--------|-----------|
| Seed script approach | Separate new script, extend existing | Rewrite existing | Current script only creates 1 org; need complete rewrite for 5+ orgs |
| Config output format | JSON, JS module, env file | JS ES module | Directly importable by vanilla JS playground app |
| Startup orchestration | Makefile, shell script, npm script | Shell script called by npm | Portable, can check health, handle signals |
| 2FA enrollment method | SQL inserts, service calls | Service calls | Uses existing validated business logic |
| Number of orgs | 2, 3, 5 | 5 | Covers all 2FA policy variants plus consent testing |
