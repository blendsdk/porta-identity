### Magic Link Flow

> Full step-by-step flow for AUTH-02 magic link authentication.

1. User enters email on login page and clicks "Send magic link"
2. Server creates a `magic_links` record with `interaction_uid` binding, sends email
3. User opens email and clicks the magic link (`GET /interaction/:uid/magic/:token`)
4. Server verifies: token is valid, not expired, not used → atomically marks as used
5. **If MFA is enabled** → render MFA challenge page within the same interaction
6. **If MFA passed or not enabled** → complete the oidc-provider interaction and redirect to the authorization callback
7. **If interaction has expired** (10 min TTL < 15 min magic link TTL) → show "session expired" page with a link to restart the login flow

**Timing edge case:** Magic link TTL (15 min) exceeds Interaction TTL (10 min). If the user opens the email after 11 minutes, the magic link is still valid but the interaction has expired. This is an accepted edge case — the user sees a "session expired" page and must restart the flow. To minimize this, the magic link email should encourage prompt action.

### Organization Member Roles (`org_role`)

The `org_role` column on `organization_members` accepts values: `owner`, `admin`, `member`.

**MVP behavior:** In MVP, `org_role` is stored as **metadata only** — it is not used for authorization checks. Phase 2 (ORG-10: org admin delegation) will add role-based authorization logic that uses these values.

**Business rules (enforced in MVP):**
- Default role for new members: `member`
- **At least one `owner` per organization** — removing the last owner or changing their role is rejected with HTTP 422
- Org role changes are tracked in the audit log (`admin.member_role_changed`)
- Bootstrap creates the initial admin user as `owner` of the `porta-system` organization

### Organization & Application Status Enforcement

> Defines WHERE in the authentication flow suspended organizations (ORG-08) and disabled applications (APP-04) are checked.

**Enforcement points (checked in order during an authorization request):**

| Check Point | When | What | Behavior on Failure |
|-------------|------|------|---------------------|
| 1. Client lookup (adapter) | oidc-provider loads client metadata | Resolve `oidc_clients` → `organization_id` → check `organizations.status` | Return `access_denied` error to oidc-provider; show "access unavailable" interaction page |
| 2. Client lookup (adapter) | oidc-provider loads client metadata | Resolve `oidc_clients` → `application_id` → check `applications.status` | Return `access_denied` error to oidc-provider; show "application unavailable" interaction page |
| 3. `findAccount` | oidc-provider resolves the authenticated user | Check `users.status` is `active` (not `suspended` or `locked`) | Return `null` from findAccount; oidc-provider treats as "no account found" |
| 4. Interaction middleware | Before rendering login/consent page | Re-check org + app status (in case changed during interaction) | Redirect to error page |
| 5. Client lookup (adapter) | oidc-provider loads client metadata | Verify `organization_applications` record exists and `enabled = true` for the client's (`organization_id`, `application_id`) pair | Return `access_denied` error to oidc-provider; show "application not available for this organization" interaction page |

**Client Credentials flow:** Steps 1-2 and 5 apply. Step 3 does not apply (no user). If org is suspended, app is disabled, or org-app subscription is inactive, token issuance is rejected.

**Admin API:** Admin API endpoints do NOT check org/app status — admins can manage suspended orgs and disabled apps. Status enforcement is only on OIDC protocol flows.

### Scope Catalog

> Complete list of OIDC scopes supported by Porta, including custom scopes. Configured in oidc-provider's `scopes` and `claims` settings.

| Scope | Type | Claims Returned | Priority | Notes |
|-------|------|-----------------|----------|-------|
| `openid` | Standard | `sub` | **MVP** | Required for all OIDC flows |
| `profile` | Standard | `name`, `given_name`, `family_name`, `picture`, `locale`, `updated_at` | **MVP** | See supported claims note below |
| `email` | Standard | `email`, `email_verified` | **MVP** | |
| `phone` | Standard | `phone_number` | **MVP** | Maps to `phone` column |
| `app:roles` | Custom | `roles`, `permissions` | **MVP** | Application-scoped roles and their resolved permissions for the current (org, app) |
| `app:organization` | Custom | `org_id`, `org_name` | **MVP** | Organization context of the authenticated user for the current client's org |
| `offline_access` | Standard | *(triggers refresh token issuance)* | **MVP** | Required for refresh token grant |

**Supported `profile` claims vs OIDC standard:**
Porta supports a subset of the standard OIDC `profile` scope claims. The following mapping applies:

| OIDC Claim | Porta Column | Supported | Notes |
|------------|--------------|-----------|-------|
| `name` | `display_name` | ✅ | |
| `given_name` | `first_name` | ✅ | |
| `family_name` | `last_name` | ✅ | |
| `picture` | `avatar_url` | ✅ | |
| `locale` | `locale` | ✅ | User's preferred language |
| `updated_at` | `updated_at` | ✅ | Unix timestamp of last profile update |
| `nickname` | — | ❌ | Not stored; `name` is sufficient |
| `preferred_username` | — | ❌ | Email is the login identifier |
| `profile` | — | ❌ | No profile URL |
| `website` | — | ❌ | Not a user attribute |
| `gender` | — | ❌ | Not collected |
| `birthdate` | — | ❌ | Not collected |
| `zoneinfo` | — | ❌ | Not stored |
| `middle_name` | — | ❌ | Not stored |

**Claim resolution rule for multi-org users:**
The organization and application context for custom claims (`app:roles`, `app:organization`) is **ALWAYS** derived from the authenticating OIDC client's `organization_id` and `application_id`. The user does not select an organization during authentication — the client determines the context. If the user is not a member of the client's organization, `app:roles` returns empty arrays (`roles: []`, `permissions: []`), and `app:organization` still returns the org details (since the org context comes from the client, not from user membership).

**Scope behavior by grant type:**
- **Authorization Code / Refresh Token:** All scopes available. `app:roles` and `app:organization` resolve from the client's (organization, application) and the user's role assignments.
- **Client Credentials:** Only `app:roles` is meaningful. Roles come from the client's `client_roles` configuration (see §4.8 CLIENT-07). `app:organization` is auto-populated from the client's `organization_id`. No user context — `openid`, `profile`, `email`, `phone` are not applicable.

### Introspection Response

> Resource servers call `/introspect` to validate opaque access tokens. The response includes standard RFC 7662 fields plus Porta custom claims.

**Standard fields always included:**
- `active` (boolean), `sub`, `client_id`, `iss`, `exp`, `iat`, `scope`, `token_type`

**Custom claims included when the corresponding scope was granted:**

| Scope Granted | Extra Introspection Fields | Notes |
|---------------|---------------------------|-------|
| `app:roles` | `roles: string[]`, `permissions: string[]` | Resolved for the client's (org, app) |
| `app:organization` | `org_id: string`, `org_name: string` | From the client's organization |
| `email` | `email: string`, `email_verified: boolean` | |

**Client Credentials introspection:** Returns `active`, `client_id`, `iss`, `exp`, `iat`, `scope`, `token_type`, plus `org_id`, `org_name`, `roles`, `permissions` (no `sub` — no user).

| I18N-03 | JSON-based translation files | **MVP** | Locale files at `locales/{lang}.json` with namespaced keys (e.g., `login.title`, `mfa.prompt`) |
| I18N-04 | Supported languages: English + Dutch | **MVP** | `en` and `nl` locale files shipped with the application |
| I18N-05 | Translatable email templates | **MVP** | Magic link, invitation, password reset, and email verification emails rendered in the user's preferred language |
| I18N-06 | Translatable server validation messages | **MVP** | Form errors and user-facing feedback messages on interaction pages are translated |
| I18N-07 | Default language configuration | **MVP** | `DEFAULT_LOCALE` env var (default: `en`). Fallback when no other locale source is available |
| I18N-08 | User locale preference | **MVP** | `locale` column on `users` table; updatable via Admin API and self-service profile (Phase 2) |
| I18N-09 | OIDC `ui_locales` parameter support | **MVP** | oidc-provider passes `ui_locales` from the authorization request; interaction pages honor it as the highest-priority locale source |
| I18N-10 | Organization default locale | **Phase 2** | Organizations can set a `default_locale` in their settings JSONB; used as fallback before the global `DEFAULT_LOCALE` |
| I18N-11 | Additional language packs | **Future** | Support for adding new languages by dropping a `locales/{lang}.json` file |

**Language Resolution Order (highest to lowest priority):**
1. OIDC `ui_locales` request parameter (set by the relying party in the authorization request)
2. User's stored `locale` preference (from `users.locale` column)
3. Browser `Accept-Language` header
4. Organization's `default_locale` setting (Phase 2)
5. Global `DEFAULT_LOCALE` env var (default: `en`)

**Scope of i18n:**
- ✅ Interaction pages (login, consent, MFA, invitation acceptance)
- ✅ Email templates (magic link, invitation, password reset, email verification)
- ✅ Server-side validation/error messages shown to users on interaction pages
- ❌ Admin API responses (English-only, machine-consumed)
- ❌ Internal logging and audit log entries (English-only)
- ✅ Consent page scope descriptions (e.g., "This application wants to access your email address")
- ❌ OIDC protocol responses (standardized by spec)

### Email Templates

> All transactional emails sent by Porta. Templates are EJS files stored at `views/emails/{template-name}.ejs`. Each template receives the resolved locale and renders in the user's preferred language using the `t('key')` helper (same as interaction pages).

| Template | Trigger | Subject Line Pattern | Available Variables |
|----------|---------|---------------------|---------------------|
| `magic-link` | AUTH-02: User requests magic link | `"Your login link for {app_name}"` | `magic_link_url`, `app_name`, `app_logo_url`, `expires_in_minutes`, `user_email` |
| `invitation` | INVITE-02: Admin sends invitation | `"You've been invited to {org_name} on {app_name}"` | `invitation_url`, `org_name`, `app_name`, `app_logo_url`, `inviter_name`, `custom_message`, `expires_in_hours`, `role_names[]` |
| `password-reset` | SELF-03: User requests password reset | `"Reset your password for {app_name}"` | `reset_url`, `app_name`, `app_logo_url`, `expires_in_minutes`, `user_email` |
| `email-verification` | SELF-02: Email verification sent | `"Verify your email address"` | `verification_url`, `app_name`, `app_logo_url`, `expires_in_hours`, `user_email` |

**Template conventions:**
- All templates include application branding (logo, colors) when resolvable from the triggering context
- Subject lines are translated via i18n keys (e.g., `email.magic_link.subject`)
- Templates have both HTML and plain-text versions (EJS renders both)
- The `{ISSUER}` base URL is used to construct all links (e.g., `{ISSUER}/interaction/{uid}/magic/{token}`)
- Sender address: `SMTP_FROM` env var for all emails
