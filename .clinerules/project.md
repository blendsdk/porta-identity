## Coding Conventions

### Naming

- **Files:** kebab-case (e.g., `magic-link.ts`, `signing-keys.ts`)
- **Classes:** PascalCase (e.g., `OidcAdapter`, `AuditLogger`)
- **Functions/Methods:** camelCase (e.g., `findAccount`, `rotateSigningKey`)
- **Constants:** UPPER_SNAKE_CASE (e.g., `MAX_FAILED_LOGINS`, `TOKEN_TTL`)
- **Types/Interfaces:** PascalCase (e.g., `Organization`, `UserOrgAppRole`)

### Key Architecture Decisions

- **No BlendSDK/WebAFX** — Pure Koa (ADR-002)
- **No DI container** — Services wired via closures/modules
- **Direct library usage** — `pg`, `ioredis`, `nodemailer` directly (no abstract providers)
- **Opaque access tokens** — Resource servers must use `/introspect` (ADR-005)
- **Signing keys in PostgreSQL** — Encrypted at rest with AES-256-GCM (ADR-007)
- **Client secrets hashed** — SHA-256, shown once at creation (ADR-009)
- **Admin API dual auth** — API key (`X-API-Key`) + admin JWT (`Authorization: Bearer`) (ADR-008)
- **SMTP for email** — Nodemailer, no SaaS dependency (ADR-006)
- **i18n** — JSON locale files, OIDC `ui_locales` parameter support, user locale preference

### Internationalization (i18n)

- **Supported languages:** English (`en`) + Dutch (`nl`)
- **Translation files:** `locales/{lang}.json` with namespaced keys (e.g., `login.title`, `mfa.prompt`)
- **Language resolution order:** OIDC `ui_locales` → user `locale` preference → `Accept-Language` header → org `default_locale` (Phase 2) → `DEFAULT_LOCALE` env var
- **Scope:** Interaction pages, email templates, server validation messages (Admin API stays English-only)
- **EJS integration:** `t('key')` helper function available in all templates
- **Config:** `DEFAULT_LOCALE` env var (default: `en`)

### Security Principles

- PKCE enforced for ALL authorization code flows
- Argon2id for password hashing
- Constant-time comparison for all tokens/secrets
- Rate limiting on sensitive endpoints
- User enumeration prevention (same response for valid/invalid emails)
- CSP headers on interaction pages
- Input validation with schema validation (zod or similar)

## Special Rules (Project-Specific)

```
- oidc-provider is Koa-based — all middleware and routes use Koa context (ctx), not Express req/res
- OIDC protocol endpoints are provided by oidc-provider — we configure, not implement
- Multi-tenant: ALL data queries must include organization_id where applicable
- Tokens: access tokens are OPAQUE (not JWT) — ID tokens are JWT per OIDC spec
- Interaction pages are server-rendered (EJS) — no SPA framework
- Per-application branding on interaction pages (resolved from client → application)
- Environment-based config only — no config files, all via env vars
- Stateless application — all state in PostgreSQL/Redis (for horizontal scaling + blue-green deploy)
- Interaction pages and email templates are multi-language (i18n) — see §4.13 in REQUIREMENTS.md
- Translation keys use namespaced format: `{page}.{element}` (e.g., `login.title`, `mfa.enter_code`)
- Admin API responses are always English — no i18n for machine-consumed endpoints
- Key reference document: plans/REQUIREMENTS.md (full requirements, data model, API surface, ADRs)
```
